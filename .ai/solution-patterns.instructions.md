## Layout choice

This repo uses **Layout A â€” by-type (flat `Controls/`)** from Â§3 of `wpf-winui-patterns.md`.

| Item | Value |
|---|---|
| Shell XAML | `Tool/MainWindow.xaml` |
| UI controls folder | `Tool/Controls/` (flat â€” no `Views/{Feature}/` subfolders) |
| MVVM separation | None â€” there is no `ViewModels/` folder; data binds directly to the static `Global.AppData` / `Global.AppSettings` and to `DataItem` instances via `INotifyPropertyChanged`. |

**Why:** the application is a small single-window tool (four tabs, two reused list controls). Introducing an MVVM layer for ~3 user controls would be more ceremony than the project warrants. The static `Global` singleton + `SettingsData<AppData>` pattern is intentional and shared across other JocysCom desktop tools.

Breadcrumbs are derived from `<TabItem>` entries in `MainWindow.xaml` whose tab body is a `<controls:* />` user control reference. The four tabs and their backing controls are:

| Tab | XAML | User control | Breadcrumb |
|---|---|---|---|
| Certificates | `MainWindow.xaml` `<TabItem>` | `Controls/DataListControl.xaml` (with `DataType="Certificates"`) | `MainWindow > Certificates` |
| Domains | `MainWindow.xaml` `<TabItem>` | `Controls/DataListControl.xaml` (with `DataType="Domains"`) | `MainWindow > Domains` |
| Options | `MainWindow.xaml` `<TabItem>` | `Controls/OptionsControl.xaml` | `MainWindow > Options` |

## Overrides

### `wpf.layout = LayoutA-flat-controls`

- `Tool/Controls/` is the only location for user-facing controls â€” there is no `Views/` or `ViewModels/` folder.
- **Why:** see [Layout choice](#layout-choice). The repo size does not justify MVVM scaffolding.
- **Recommended future direction:** if the tool grows to require independent test-able view-models for cert/domain scanning state, introduce `Tool/ViewModels/` and migrate `DataListControl` to MVVM at that time (Layout B). Until then, treating the existing flat layout as "on convention" avoids busywork.

### `wpf.reused-control = DataListControl serves both Certificates and Domains`

- `Tool/Controls/DataListControl.xaml` is referenced twice from `MainWindow.xaml` â€” once per tab â€” and is parameterised by a `DataType` dependency property (`Certificates` or `Domains`).
- **Why:** the two tabs share almost identical column sets, sorting, scan triggering and result rendering. Duplicating the XAML into `CertificatesListControl` + `DomainsListControl` would have produced two near-identical XAML files and two near-identical code-behind classes.
- **Implication for the deviation report:** the row for `Tool/Controls/DataListControl.xaml` has an inferred breadcrumb of `MainWindow > DataList` (from the file name), but the *actual* breadcrumbs are `MainWindow > Certificates` and `MainWindow > Domains`. This is **accepted** â€” the control is intentionally reused. A future `pattern_map.py` revision may model multi-mount controls; until then, the CSV value is informational, not a defect.

### `wpf.automation-ids = absent throughout`

- No `AutomationProperties.AutomationId` is present in any XAML in this repo (`Tool/Controls/*.xaml`, `Tool/MainWindow.xaml`, or any vendored `Tool/JocysCom/Controls/*.xaml`).
- **Why:** there is no test project (see [`testing.absent`](#testingabsent) below) and no UI Automation consumer downstream â€” so AutomationIds would be dead code maintained for no current consumer.
- **Recommended future direction:** when a test project is introduced (per the [`testing.absent`](#testingabsent) plan), add AutomationIds with `Screen.Element` prefixes from the table in [Layout choice](#layout-choice) â€” e.g. `Certificates.Scan`, `Options.ValidFromRegex` â€” as part of the same change set, not as a retrofit.
- **Deviation policy:** rows whose Role is `ui-view` or `shell` will be reported as `off-convention` by the validator. This is **accepted** for now; the absence is declared.

### `testing.absent = no Tests project in this solution`

- There is no `*.Tests.csproj` anywhere in the repo. The `Tool.Tests/` path the validator computes for every `ExpectedTestPath` does not exist as a directory at all.
- **Why:** the project is small, has no consumers other than its end users, ships infrequently (1.0.9 â†’ 1.1.6 over ~3 years), and has never had a test project. Adding one purely to silence `test-missing` rows would not earn its keep.
- **Implication for the deviation report:** every code row reports `Deviation = test-missing`. This is **accepted en masse** â€” do not file individual issues for `test-missing` rows on this repo.
- **Recommended future direction:** if/when tests are introduced, create `Tool.Tests/Tool.Tests.csproj` (or rename `Tool/` to `JocysCom.SslScanner.Tool/` and the test project to `JocysCom.SslScanner.Tool.Tests/`) and add the `[Generator]Tests.cs` files predicted by the CSV. The `Tool/Common/` types (`DataItem`, `AppData`, the `Test_SSL_Support` static helper) are the most natural starting points: they're pure-CLR and require no UI Automation.

### `wpf.vendored-source = JocysCom shared library and ThirdParty present in-project`

- `Tool/JocysCom/` is a copy of the `JocysCom.ClassLibrary` shared library, vendored as source rather than referenced via NuGet or `<ProjectReference>`.
- `Tool/ThirdParty/` is third-party code (Whois.NET-derived) vendored the same way.
- **Why:** the shared library is maintained by the same author and intentionally co-located with each consuming tool to keep releases self-contained and signature-stable. Whois.NET is vendored because the upstream package has been unmaintained.
- **Implication for the deviation report:** rows under `Tool/JocysCom/` and `Tool/ThirdParty/` are **scanned** by `pattern_map.py` and will produce `Role`, `ExpectedNavPath` (for the few vendored XAML files), and `ExpectedTestPath` columns. The `test-missing` rows for these paths are **doubly accepted**: per [`testing.absent`](#testingabsent), and because vendored code's tests live in the upstream repository â€” not here. Do not propose moves or renames inside `Tool/JocysCom/` or `Tool/ThirdParty/`; they are mirrored.

### `wpf.shared-icons-namespace = Tool/Resources/Icons + Tool/JocysCom/Controls/Themes`

- Icon resource dictionaries live in two places:
  - `Tool/Resources/Icons/Icons_Default.xaml` â€” app-specific icons (`Icon_environment_network`).
  - `Tool/JocysCom/Controls/Themes/Icons.xaml` â€” shared library icons (`Icon_lock`, `Icon_environment`, `Icon_gearwheel`, `Icon_Information`).
- **Why:** the shared icons travel with the vendored library; the app-specific icons stay with the app. `MainWindow.xaml` references icons from both via `{StaticResource}` lookup against the merged resource dictionary.
- **Implication for the deviation report:** both files report `Role=resource` and have no `ExpectedNavPath` / `ExpectedTestPath`. No deviation is expected or accepted.
