# Solution Patterns — declared for this repo

This file declares the structural conventions in force in the JocysCom SslScanner repository. It overrides the default rules shipped in [`.ai/skills/solution-patterns/references/`](skills/solution-patterns/references/). The accompanying [`solution-patterns.csv`](solution-patterns.csv) is generated by [`pattern_map.py`](skills/solution-patterns/scripts/pattern_map.py) and must not be hand-edited.

## Detected stacks

| Stack | Evidence | Reason |
|---|---|---|
| `wpf` | `Tool/JocysCom.SslScanner.Tool.csproj` | `<UseWPF>true</UseWPF>` |
| `winforms` | `Tool/JocysCom.SslScanner.Tool.csproj` | `<UseWindowsForms>true</UseWindowsForms>` (interop only — no `*Form.cs` Windows Forms designer files exist) |

The project also targets `net8.0-windows` and outputs `WinExe`. There is no ASP.NET Core, Angular, Next.js, or SQL Data project in this repository.

## SSOT directions

No name-mapping correspondences apply in this repo.

- **No `sql-model` SSOT** — there is no database or SQL Data project. All persistence is XML serialisation of the `AppData` settings tree (see `.ai/repository-analysis.instructions.md` §8). The `ExpectedSqlTable` / `ActualSqlTable` columns therefore remain blank for every row.
- **No code-generation pipelines** — no T4 templates, no Roslyn source generators (other than the implicit WPF XAML compiler), no `Generate-*.ps1` script that emits C# from another source. The `manual-edit-of-generated` deviation code is not expected to fire.

## Layout convention

The single project follows **Layout A — by-type** from [`wpf-winui-patterns.md`](skills/solution-patterns/references/wpf-winui-patterns.md):

```text
Tool/MainWindow.xaml                     ← shell with TabControl
Tool/Controls/AboutControl.xaml          → MainWindow > About
Tool/Controls/DataListControl.xaml       → MainWindow > Certificates / Domains (one control, two tabs, dataType-driven)
Tool/Controls/OptionsControl.xaml        → MainWindow > Options
Tool/JocysCom/Controls/InfoControl.xaml  → header banner inside MainWindow (not a tab)
```

**Why Layout A and not Layout B (MVVM `Views/{Name}/{Name}View.xaml`):** the tool is small, has no separate view-model layer, and binds directly to `Global.AppSettings` via static-resource bindings. Adopting MVVM-style folders would create empty `ViewModels/` directories with no real residents and make the project feel larger than it is.

## Overrides

### `wpf` / `winforms` — desktop view breadcrumb derivation

- **DataListControl serves two tabs.** The single `Controls/DataListControl.xaml` is loaded twice in `MainWindow.xaml` — once for the **Certificates** tab and once for the **Domains** tab — distinguished by its `DataType` dependency property (`Certificates` | `Domains`). The CSV records its breadcrumb as `MainWindow > DataList` (the generic default); the actual user-facing labels are `MainWindow > Certificates` and `MainWindow > Domains`.
  - **Why:** the generator infers breadcrumbs from the control class name, not from the tab `Header`. A future enhancement to `pattern_map.py` could scan `MainWindow.xaml` for `<controls:DataListControl ... DataType="..."/>` instances and emit two rows; for now the divergence is recorded here.
  - **How to apply:** when adding a new screen, name the UserControl after the tab (e.g. `LogsControl.xaml` → `MainWindow > Logs`) so the generator's default works without an override.

- **`JocysCom/Controls/*` are framework primitives, not application screens.** The XAML files under `Tool/JocysCom/Controls/` (`InfoControl`, `MessageBoxWindow`, `ProgressBarControl`) are vendored library code (see `.ai/repository-analysis.instructions.md` §11). They are loaded as helpers, not as top-level menu items. The CSV's `ExpectedUiPath` of `MainWindow > Info` / `MainWindow > MessageBox` / `MainWindow > ProgressBar` is therefore misleading and should be treated as informational only.
  - **Why:** the generator can't distinguish vendored framework code from first-party UserControls today. Treat their breadcrumbs as artefacts of the default rule, not as expected navigation entries.
  - **How to apply:** do not chase those rows. If they appear in a deviation report, mark `Accepted` and leave them.

### Test layout

- **No test project exists.** The CSV currently reports `test-missing` for every code file because the qa-tester mirror rule expects `Tool.Tests/...` and that project has not been created.
  - **Why:** the repository ships only the production tool and one manual harness (`Tool/Common/Test_SSL_Support.cs` + `.bat`). There is no `MSTest` / `xUnit` / `NUnit` package reference in the .csproj.
  - **How to apply:** the `test-missing` rows are accurate but not actionable today — treat them as a backlog signal, not a defect. When a test project is added later, name it `Tool.Tests` to satisfy the default `{Project}.Tests/{...}` mirror.

### Resources, themes, third-party

- **`Tool/JocysCom/`** is a curated source-level copy of the [JocysCom.ClassLibrary](https://github.com/JocysCom/ClassLibrary) shared library. Treat its contents as upstream code — do not refactor in place; upstream changes should land in the source library and be re-vendored via `Tool/JocysCom/MakeLinks_Ref.ps1`.
- **`Tool/ThirdParty/`** vendors Whois.NET (`WhoisClient.cs`, `WhoisResponse.cs`) and an `IPAddressRange` helper. Same rule: do not refactor in place; report upstream if a fix is needed.
- **`Tool/Resources/Icons/`** is a first-party SVG → XAML icon dictionary. New icons go through `Icons_Default.SVG_to_XAML.ps1`.

## Companion rules

| Companion pair | Status in this repo |
|---|---|
| `.xaml` ↔ `.xaml.cs` (code-behind) | **All present.** Every `.xaml` in the project has its `.xaml.cs` sibling — including the vendored `JocysCom/Controls/*` files and the `Resources/Icons/Icons_Default.xaml`. No `companion-missing` deviations expected. |
| `View.xaml` ↔ `ViewModel.cs` (MVVM pairing) | **Not used.** This project does not follow MVVM separation; UserControls hold their own logic in code-behind and bind directly to `Global.AppSettings`. |

## How to read the CSV

- `Role=resource` rows (`Themes/Default.xaml`, `Themes/Icons.xaml`, `Resources/Icons/Icons_Default.xaml`) have blank `ExpectedUiPath` and blank `ExpectedTestPath` and are correctly reported as `none`.
- `Role=bootstrap` (`App.xaml`) and `Role=shell` (`MainWindow.xaml`) have no UI breadcrumb but do have an expected test path.
- `Role=ui-view` rows under `Controls/` correspond to actual tabs; rows under `JocysCom/Controls/` correspond to vendored helper windows (see the override above).
- `Role=converter` (`ItemFormattingConverter`, `TabIndexConverter`) and `Role=code-behind` rows are unit-testable in principle.
- All `test-missing` rows reflect the absence of a `Tool.Tests` project, not per-file defects.

```

To re-run the deviation report:

```

Both scripts are idempotent. Commit the regenerated CSV alongside the code change that motivated it.
